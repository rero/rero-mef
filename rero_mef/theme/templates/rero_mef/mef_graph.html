How MEF Linking Works

MEF (Multilingual Entity File) merges authority records from multiple national library sources into a single unified record per entity. The linking strategy differs by entity type — agents use VIAF as a central hub, while concepts and places rely on shared identifier codes embedded directly in the source records.

Agents Persons · Organisations

Agent records from GND (Deutsche Nationalbibliothek), IDREF (Agence bibliographique de l’enseignement supérieur), and RERO (Réseau des bibliothèques de Suisse occidentale) are linked through VIAF (Virtual International Authority File) — an international hub that cross-references authority files from dozens of national libraries.

Each source record carries a VIAF source code inside its identifiedBy list. VIAF groups records from different sources that refer to the same person or organisation under a shared cluster. MEF harvests those VIAF clusters and uses the shared viaf_pid to merge the source records into one unified MEF record.

Sources GND IDREF RERO
Linking hub VIAF
MEF link key viaf_pid
VIAF source codes DNB (GND) · FRBNF (IDREF) · RERO (RERO)

Concepts Topics · Subjects · Genres

Concept records from GND and IDREF are linked through a shared BNF identifier (Bibliothèque nationale de France). There is no external hub — the link key is embedded directly in the source records.

IDREF concept records carry a FRBNF* number in their identifiedBy list (source BNF). GND concept records carry the same FRBNF* number inside their exactMatch entries. If no exactMatch entry contains a BNF identifier, MEF falls back to closeMatch — but only when exactly one match is found, to avoid ambiguous links. MEF compares these codes across both sources to find and merge matching records.

RERO concept records are also included in MEF but do not carry a cross-source identifier, so they are not directly linked to GND or IDREF records.

Sources GND IDREF RERO (RERO not cross-linked)
Linking method BNF association identifier
Link key FRBNF* number
IDREF field identifiedBy[source="BNF"].value
GND field exactMatch[].identifiedBy[source="BNF"].value
fallback: closeMatch[] when exactly one match

Places Geographic locations

Place records from GND and IDREF are linked through the GND PID. RERO does not publish place authority data, so there is no RERO source for places.

GND place records serve as the anchor: their own pid is the link key. IDREF place records embed the corresponding GND PID (prefixed with (DE-101)) inside their identifiedBy list (source GND, type bf:Nbn). MEF strips the prefix and matches the bare PID to find the corresponding GND record.

Sources GND IDREF
Linking method GND PID cross-reference
GND link key own pid (anchor)
IDREF link key identifiedBy[source="GND", type="bf:Nbn"].value
e.g. (DE-101)1234567

Record redirects & lifecycle How superseded records are tracked

Source authorities regularly merge or deprecate records. MEF tracks these changes through a relation_pid field on the source record embedded in each MEF record. The type depends on the source:

SourceTypeMeaning
GND redirect_to Old GND record (MARC 682 “Umlenkung”) points to its canonical replacement. The old record has its own MEF entry.
IDREF redirect_from Canonical IDREF record (MARC 035 sudoc) records the old SUDOC PID it replaced. The old PID may have no separate MEF entry.

The latest API endpoint resolves any source PID to the current MEF record:

Agents /api/agents/mef/latest/<pid_type>:<pid>
Concepts /api/concepts/mef/latest/<pid_type>:<pid>
Places /api/places/mef/latest/<pid_type>:<pid>

Example: /api/agents/mef/latest/gnd:069774331 follows the redirect_to chain to return the current MEF record.

In the record detail view, the field used as link key is highlighted with a link key badge in the identifiedBy section.
